## site/i18n.js (NEW — a leaf ES module, imports nothing)

Exports:
- `export const LANGS = ['en', 'it'];`
- `export const STRINGS = { en: { …keyed strings… }, it: { …keyed strings… } };`
  - Values are **author-controlled literals only** — never user input. `STRINGS.en` mirrors the
    shipped English exactly. Keys are short and semantic (dot-namespaced), e.g. `nav.cv`,
    `hero.intro`, `hero.type.lead`, `mission.label`, `mission.line`, `origin.h2`, `origin.beat1`,
    `work.plate1.title`, `exp.row1.role`, `method.h2`, `calc.mash`, `dd.cue.diverge`, …
- `export function getLang()` — current language (defaults 'en' before first applyLang).
- `export function getInitialLang()` — `localStorage.getItem('sirio-lang')` if it is one of LANGS,
  else `navigator.language && navigator.language.toLowerCase().startsWith('it') ? 'it' : 'en'`.
- `export function t(key)` — returns `STRINGS[getLang()][key] ?? STRINGS.en[key] ?? key`
  (EN fallback so a missing IT key never blanks the page).
- `export function onLangChange(cb)` — pushes `cb` into a module-level `Set`; dynamic modules
  subscribe so they re-render on switch.
- `export function applyLang(lang)`:
  1. clamp `lang` to LANGS (fallback 'en'); store as current.
  2. `document.documentElement.lang = lang;`
  3. `document.title = t('meta.title');` (kept "Sirio Feltrin" in both — routed through a key so
     everything is centralized; add other head keys only if a `<meta name="description">` is added).
  4. For every `[data-i18n]` element: `el.textContent = t(el.dataset.i18n);`
  5. For every `[data-i18n-rich]` element: `el.innerHTML = t(el.dataset.i18nRich);`
     (innerHTML from an author-controlled literal dictionary — same trust model as the literal
     strings in method.js; see `<threat_model>`.)
  6. For every `[data-i18n-attr]` element: value is `"attr:key; attr2:key2"`; split on `;`, each on
     the first `:`, and `el.setAttribute(attr.trim(), t(key.trim()))`. Used for `aria-label`,
     `alt`, and the media `data-label` placeholders.
  7. `localStorage.setItem('sirio-lang', lang);`
  8. `subs.forEach(cb => cb(lang));`

**Annotation rule (applies to index.html):** put the attribute on the INNERMOST element that
wraps only translatable text. Never replace an element whose child `<span>` is required by a
reveal/assemble animation — key that inner `<span>` instead (many `.narr` beats are
`<p ...><span>text</span></p>`: key the `<span>` with `data-i18n`). Use `data-i18n-rich` ONLY when
the translatable text itself contains inline markup (`<span class="grad">`, `<em>`, `<a>`,
`<strong>`) and include that markup verbatim in the dictionary value. Everything else uses plain
`data-i18n` (textContent). Preserve `&mdash;`/`—`, `·`, `&amp;` exactly, in both languages.

## Wiring (main.js boot)

Import `{ applyLang, getInitialLang, onLangChange, t }` from `./i18n.js`. In `boot()`:
- Register subscribers BEFORE the first applyLang:
  - `onLangChange((lang) => star.setLang(termsFor(lang)))` where `termsFor(lang)` builds the term
    array from the dictionary (see below). (method.js registers its own subscriber via initMethod.)
- Wire the toggle: `document.querySelectorAll('.nav__lang-btn').forEach(b => b.addEventListener('click', () => { applyLang(b.dataset.lang); syncLangButtons(b.dataset.lang); }));`
  where `syncLangButtons` sets `aria-pressed` true/false on the two buttons.

  `initReveals()` and `initMethod()` have run (so their elements exist and their subscribers are
  registered). This is inside `boot()`, which already runs after `document.fonts.ready`.

## Hero typewriter → language-aware (star-engine.js, TEXT ONLY)

Keep the gradient palette in star-engine (design, not language). Change the term SOURCE:

  existing gradients. EN supplies `text` only (existing vowel rule computes the article). IT
  supplies `text` **and an explicit `prefix`** per word (the correct Italian article incl. spacing,
  e.g. `"un "`, `"una "`, `"uno "`, or `""` where an article is dropped) so no vowel-rule guessing.
- Add `HeroStar.setLang(termList)`: accepts the resolved array `[{text, grad, prefix?}, …]`,
  rebuilds `this.terms` (applying `t.prefix ?? (/^[aeiou]/i.test(t.text) ? 'n ' : ' ')` and
  `t.full = t.prefix + t.text`), then `clearTimeout(this._tt); this.ti = 0; this.ci = 0;
  this.deleting = false;` re-seeds the first gradient, and (if not reduced) calls `this._tick()`.
  It touches NOTHING else — no `periodMs`, `t0`, breath, `armed`, `orbiting`, `planets`.

## Method calculator → language-aware (method.js)

- The tracks' STATIC prose (track kind, `track__for`, `track__title`, phase names/descs/note,
  doc-rail, `track__foot`, evidence, deck-link) is handled by `data-i18n` in index.html — it updates
  automatically via applyLang, no method.js change needed for those.
- The calc's DYNAMIC strings are injected by method.js timers. Move each literal into the dictionary
  and read via `t('calc.*')` at play time (so a fresh play uses the current language). Keys:
  `calc.ghost.q` ('873 ÷ 41 = ?' — keep the math), `calc.mash` ('ANSWER MATCHED. UNDERSTOOD
  NOTHING.'), `calc.order.head` ('THEN I LEARNED THE ORDER'), `calc.line1/2/3`
  ('MAP THE PROBLEM' / 'CHOOSE THE STRATEGY' / 'LET THE TOOL EXECUTE'), `calc.reverse`
  ('NEVER THE REVERSE'), `calc.direction` ('THE DIRECTION IS MINE'), `calc.reduced.ghost`
  ('MAP → STRATEGY → EXECUTE'). Numbers ('0', '21.29268') stay literal.
- Subscribe method to `onLangChange`: on change, `timers.forEach(clearTimeout)` then snap the calc
  to its RESOLVED localized end-state (exactly the REDUCED branch of `playCalc`: ghost =
  `t('calc.reverse')` or `t('calc.reduced.ghost')`, out `--resolved` = `t('calc.direction')`). This
  means a mid-page language switch does NOT restart the 20s sequence in the wrong language — it just
  re-labels the resolved state. If the sequence has not started yet (not scrolled into view), the
  play-once observer will play it in the current language when it fires (playCalc reads `t()` live).
