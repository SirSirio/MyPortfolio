**Researched:** 2026-07-18
**Domain:** Vanilla-JS scroll-telling (GSAP + ScrollTrigger) inside a ported Canvas warp overlay, on a static GitHub Pages site

## Summary

This phase ports an already-written warp/hyperlight transition system from the V4 design snapshot into the live `site/`, adds ONE `[data-proj="ot2"]` overlay page reached by making the existing OT-2 card clickable, and drives its scroll-telling with GSAP + ScrollTrigger. Nearly everything the phase needs already exists in the repo: the warp machinery is complete in `Sirio V4 - Deep Field x Hyperlight.dc.html:266-520`, the timelapse is already re-encoded to 1.96 MB with a poster in `site/assets/media/`, the four CAD renders are already committed, the `REDUCED` reduced-motion flag and `--accent` section system are established, and the content source (the DALSA report) extracts cleanly with the `pdftotext` already on this machine. The work is integration and taste, not greenfield invention.

Three things needed real investigation and are answered below. **R-03 (GSAP integration):** GSAP 3.15.0 is current, fully free (MIT-equivalent "no charge" license, all plugins free since 3.13), and all four CDN URLs resolve — but the site's own media contract forbids hot-linking remote `src`, so the plan-safe recommendation is to **vendor** the two `.min.js` files into `site/assets/vendor/`. The overlay is its OWN scroll container (`overflow:auto`), so every ScrollTrigger must set `scroller` to that element, not the window. **R-02 (vendor images):** neither Opentrons nor Embi Tec publishes any press-kit or media-use grant permitting editorial reproduction of their marketing renders — so `Opentrons-OT2.png` and `RunOne Gel Electrophoresis System.png` must be treated as **not cleared**; fall back to Sirio's own bench photo or line-art (D-16). **R-01 (scroll-telling patterns):** the cheap-on-mobile vocabulary is transform/opacity tweens + SVG `stroke-dashoffset` draw-on, `scrub`-linked to the overlay scroll, with pinning used sparingly.

**Primary recommendation:** Port the warp system verbatim into `site/main.js` (adding the `curProj` engine-pause guard the site loop currently lacks), vendor GSAP+ScrollTrigger, scope all ScrollTriggers to the `[data-proj="ot2"]` scroller inside a `gsap.context()` that is reverted on close, gate every tween on the existing `REDUCED` import (no second media query), and build the page content from the DALSA report + profile.md. Do NOT ship the vendor marketing renders.

### Pattern 1: ScrollTrigger scoped to the overlay scroller (NOT window)
**What:** the `[data-proj]` overlay is `position:fixed; inset:0; overflow:auto` — it scrolls internally; `window.scrollY` does not change while it is open. Every ScrollTrigger must therefore point `scroller` at that element.
**When to use:** every scroll-linked tween on the deep-dive page.
```javascript
// Source: gsap.com/docs/v3/Plugins/ScrollTrigger — `scroller` property
gsap.registerPlugin(ScrollTrigger);
const scroller = document.querySelector('[data-proj="ot2"]');  // the overflow:auto element
gsap.to('.blueprint-line', {
  strokeDashoffset: 0,
  scrollTrigger: {
    scroller,                       // <-- critical: overlay, not viewport
    trigger: '.build-section',
    start: 'top 80%',
    end: 'bottom 60%',
    scrub: 1                        // ties progress to scroll; reverses on scroll-back
  }
});
```

### Pattern 2: Draw-on with CSS dashoffset (no DrawSVG plugin)
**What:** animate a blueprint line/path being drawn by tweening `stroke-dashoffset` from the path length to 0. Cheap on mobile; GPU-friendly; no extra plugin file.
```javascript
// Source: gsap.com community — stroke-dasharray/offset draw-on
const path = document.querySelector('.blueprint__trace');
const len = path.getTotalLength();
path.style.strokeDasharray = len;
path.style.strokeDashoffset = len;               // hidden start
gsap.to(path, { strokeDashoffset: 0, scrollTrigger: { scroller, trigger: path, scrub: true } });
```
Mobile caveat (R-01): Safari mishandles NEGATIVE dashoffset — keep the offset positive (start = +len, end = 0). Avoid `filter: saturate()` on animated SVG (high GPU cost).

### Pattern 3: Lifecycle — build on arrival, revert on close (gsap.context)
**What:** create all deep-dive triggers inside a `gsap.context()` so a single `.revert()` kills every tween AND every ScrollTrigger and restores inline styles on close — no leaked triggers, no double-registration if reopened.
```javascript
let ot2Ctx = null;
function initOt2Motion() {
  if (REDUCED) return;                              // reduced-motion: never build motion (see Pattern 4)
  const scroller = document.querySelector('[data-proj="ot2"]');
  ot2Ctx = gsap.context(() => {
    /* all gsap.to(...) / timelines with scrollTrigger:{scroller,...} here */
  }, scroller);
  ScrollTrigger.refresh();                          // recompute AFTER the warp transform is cleared
}
function killOt2Motion() { if (ot2Ctx) { ot2Ctx.revert(); ot2Ctx = null; } }
```
Call `initOt2Motion()` at the END of `applyTrans` when `t >= 1` and `tr.into.id === 'ot2'` (i.e., once the warp is done and `[data-proj-inner]`'s transform/filter are cleared). Call `killOt2Motion()` from `closeProj`.

### Pattern 4: Reduced-motion — ONE source of truth
**What:** gate on the existing `REDUCED` import from `star-engine.js` (`:12`). Do NOT introduce `gsap.matchMedia()` reduced-motion handling or a second `matchMedia('(prefers-reduced-motion…)')` — that would be a second media-query source, violating 02-CONTEXT D-36 and the reveal contract.
**How:** under `REDUCED`, never build the ScrollTrigger timelines; the page's HTML renders fully by default. Mirror the site's reveal invariant: JS is the ONLY thing that hides an element, so a reduced-motion visitor (or a JS failure) sees the whole page. The warp itself already has a reduced-motion branch (`openProj`/`closeProj` at `:399-403`/`:415-417`): instant show/hide, no starfield.

### Anti-Patterns to Avoid
- **Leaving `scroller` at the window.** ScrollTriggers will never fire (window doesn't scroll while the overlay is open). The single most likely bug in this phase.
- **Building/refreshing ScrollTrigger during the warp.** `[data-proj-inner]` is scaled+blurred mid-transition; measuring then yields wrong start/end. Build after `t>=1`.
- **A second rAF loop.** The home already runs ONE loop (`main.js:207`). GSAP has its own internal ticker — do NOT add a manual `requestAnimationFrame` for GSAP, and do NOT add a second loop to the home. The overlay is the sanctioned place where GSAP owns motion, and the deep-field engine is paused there.
- **A second reduced-motion media query.** Use `REDUCED`.
- **Adding the technical accent to `PAL`.** `styles.css:18-20` warns: adding a stop to the star spectrum re-tints all background stars. A new accent is a `--accent`/section variable only.

## Common Pitfalls

### Pitfall 1: ScrollTrigger measures the window instead of the overlay
**What goes wrong:** tweens never fire, or fire at the wrong scroll positions.
**Why:** the overlay scrolls internally (`overflow:auto`); the window is frozen (`body overflow:hidden` while open).
**How to avoid:** set `scroller` to `[data-proj="ot2"]` on every trigger. Verify by scrolling the overlay and watching a tween respond.
**Warning signs:** animations only work if you resize, or fire immediately on open.

### Pitfall 2: The engine keeps drawing behind the overlay
**What goes wrong:** the deep-field + hero keep running under the open overlay, contending with GSAP for the frame budget on mobile (jank).
**Why:** the CURRENT site loop (`main.js:207-212`) calls `drawDeep` + `star.frame` unconditionally — it does NOT yet have the design's `curProj` guard (`.dc.html:352`).
**How to avoid:** port the guard: `if (!(curProj && !trans)) { drawDeep(now); applyPara(); }`. This realizes D-20's "engine paused behind overlay" invariant — reconcile it deliberately, it is the justification for adding GSAP.
**Warning signs:** frame rate drops when the deep-dive is open; battery drain on phones.

### Pitfall 3: The plate is clickable by mouse only (keyboard/A11y regression)
**What goes wrong:** the design wires a plain `<div>`/`<article>` with a click handler — not keyboard-reachable.
**Why:** the ported handler listens for `click` only.
**How to avoid:** make the OT-2 card's open-trigger a real `<button>` or add `role="button"`, `tabindex="0"`, and an Enter/Space `keydown` handler; give it an accessible label. Baseline A11y is formally Phase 5, but do not ship a NEW keyboard trap.
**Warning signs:** Tab never lands on the card; Enter does nothing.

### Pitfall 4: The report's "~10 min/gel" figure carries an author's FIX-ME note
**What goes wrong:** copying a number the author himself flagged as unverified.
**Why:** report §3.3.6 (p.28) literally reads *"The run time for loading 48 samples is about 10 minutes. -FIX THIS IN DENMARK!!-"* — an editorial reminder to re-measure.
**How to avoid:** profile.md independently states "~10 min per gel" and "144 samples/run (3 gels × 48)", so those are safe to use as Sirio's own claims. But do NOT present the figure with false precision, and if in doubt phrase as "~10 minutes". Confirm the number with Sirio if the page leans on it. (Ties to D-10: prefer the checkable, meaningful claim.)
**Warning signs:** a suspiciously exact runtime with no qualifier.

### Pitfall 5: The two vendor renders sneak onto the page
**What goes wrong:** shipping copyrighted marketing images with no use grant (see R-02).
**How to avoid:** do NOT copy `Opentrons-OT2.png` / `RunOne Gel Electrophoresis System.png` into `site/assets/media/`. Use Sirio's own bench photo or line-art for the "what I started with → what I built" contrast.
**Warning signs:** those two filenames appearing in an `<img src>`.

## Research Directive Answers

### R-01 — GSAP/ScrollTrigger scroll-telling patterns for an automation case-study, cheap on mobile
**Vocabulary that fits "automation" and stays cheap:**
- **Parts-assemble:** stagger transform (`x/y/rotation/scale`) + `opacity` tweens on the CAD-part images/SVGs as the BUILD section enters; `scrub`-linked so scrolling "assembles" the machine. Transform+opacity only = compositor-friendly, no layout thrash.
- **Protocol-trace / blueprint draw-on:** SVG `stroke-dashoffset` from path-length→0 (Pattern 2). Reads as a schematic drawing itself — ideal for the "protocol tracing itself" idea (D-13). Positive offset only (Safari), no `filter: saturate` (GPU).
- **Reveal-on-enter:** for anything that only needs to appear once, IntersectionObserver (already the site's home vocabulary) is cheaper than a ScrollTrigger; reserve GSAP for `scrub`/timeline choreography.
- **Pinning:** use sparingly. Pinning a section while an inner sequence plays is powerful but the most expensive ScrollTrigger feature on mobile (forces layout during scroll). If used, pin ONE lightweight container, not media-heavy DOM.

**Cheap-on-mobile rules (Phase 3 does the formal tuning; keep it plan-safe now):**
1. Animate only `transform` and `opacity`; avoid `top/left/width/height`, box-shadow, and SVG filters in tweens.
2. `will-change:transform` on the few animated elements only (the overlay inner already sets it).
3. Prefer `scrub` over long autonomous timelines so motion is bounded by the user's scroll, not a runaway rAF.
4. Keep total simultaneous tweens low; stagger rather than fire dozens at once.
5. The deep-field engine is paused behind the overlay (D-20) — that frees the budget GSAP needs.
`[CITED: gsap.com/docs/v3/Plugins/ScrollTrigger; GSAP community threads on dashoffset/DrawSVG mobile perf]`

### R-03 — GSAP + ScrollTrigger integration with this site
- **Version + license:** GSAP **3.15.0**, free "no charge" license; all plugins free since 3.13. `[VERIFIED: npm registry + gsap.com]`
- **CDN URLs (all HTTP 200 this session):**
  - jsDelivr: `https://cdn.jsdelivr.net/npm/gsap@3.15.0/dist/gsap.min.js` + `.../dist/ScrollTrigger.min.js`
  - cdnjs: `https://cdnjs.cloudflare.com/ajax/libs/gsap/3.15.0/gsap.min.js` + `.../3.15.0/ScrollTrigger.min.js`
- **Vendored vs CDN (GitHub Pages, no server runtime):** **recommend vendoring** the two `.min.js` into `site/assets/vendor/` and committing them. Rationale: (1) the repo's media contract explicitly forbids hot-linking a remote `src` (README:22-24) — the same principle applies to a library; (2) removes a third-party runtime dependency and point of failure; (3) version is pinned in-repo; (4) no build step needed (plain `<script>`). GSAP's free license permits self-hosting/redistribution. CDN is the documented alternative — if used, pin the exact version and add SRI `integrity` attributes.
- **Reduced-motion gating:** gate ALL GSAP motion on the existing `REDUCED` import from `star-engine.js:12`. Do NOT add `gsap.matchMedia()` reduced-motion or any second `matchMedia('(prefers-reduced-motion…)')` (02-CONTEXT D-36 — single source of truth). Under `REDUCED`, skip building the timelines entirely; the page HTML renders fully by default (mirror the reveal contract: JS is the only thing that hides).
- **ScrollTrigger scoping:** the `[data-proj="ot2"]` overlay is the scroll container (`overflow:auto`); set `scroller` to it on every trigger, NOT the window (the window is frozen via `body overflow:hidden`). `[CITED: gsap.com/docs/v3/Plugins/ScrollTrigger — scroller]`
- **Build/refresh timing:** create triggers only AFTER the warp completes (`applyTrans` `t>=1`, transform/filter cleared on `[data-proj-inner]`), then `ScrollTrigger.refresh()`. Building during the warp measures a scaled/blurred element and yields wrong positions.
- **Cleanup/kill on close:** wrap all deep-dive triggers in `gsap.context(fn, scroller)` and call `ctx.revert()` in `closeProj` — reverts inline styles and kills every tween + ScrollTrigger, so reopening rebuilds cleanly. (`ScrollTrigger.getAll().forEach(t=>t.kill())` / `ScrollTrigger.killAll()` also work but context is tidier and scoped.) `[CITED: gsap.com/docs — getAll/killAll/refresh]`
- **Engine-paused invariant:** port the `curProj` guard (`.dc.html:352`) into the site loop so `drawDeep`/`applyPara` are skipped while the overlay is open — this is what makes GSAP safe alongside the single-rAF engine (D-20). `drawWarp` still runs (it only paints during transitions).
- **Mobile performance:** transform/opacity-only tweens, `scrub` over autonomous timelines, minimal `will-change`, sparing pinning; formal mobile tuning is Phase 3.
