## Implementation Decisions

### Template scope & working model

- **D-01:** **No rigid/parameterized code template.** The `design-taste-frontend`
  skill frames each page per-brief, so a data-driven template engine would fight it
  and risk a "stamped" feel. This OT-2 page is a **hand-crafted pilot**, discussed
  in detail.
- **D-02 [informational]:** **Phase 4 working model (forward decision, not built here):** the other
  project pages get **more automation** — Sirio supplies media + content and gives
  **general style direction**; the taste-skill + AI does the framing with minimal
  per-page discussion. "Reusable template" = a proven hand-crafted *pattern/shape*,
  not a code template.
- **D-03:** **Division of labor:** Sirio owns **media + content**; Claude owns
  **style** (steered by Sirio's general directions and reactions to rendered results).
- **D-04:** **Only the OT-2 card becomes clickable** this phase (add
  `data-open="ot2"` to the Gel Electrophoresis plate, `site/index.html:214-223`).
  Other cards stay static until Phase 4.

### Page story & structure

- **D-05:** **Problem-first spine.** Open on the pain, not the payoff: gel
  electrophoresis is an everyday verification step in biotech, and loading many
  samples by hand is slow, repetitive, error-prone, and scales badly — worst exactly
  where budgets are tightest (academic labs, startups, small biotech). The gap:
  between tedious manual slab gels and expensive closed cartridge systems.
- **D-06:** **Source the problem framing from the DALSA report §1.1–1.2**
  (`06_Materials/…Report - s240104.pdf`) — it is far richer than profile.md's single
  line. Frame it well; do not copy verbatim.
- **D-07:** **Section skeleton (curated):**
  1. **PROBLEM** — the manual-loading bottleneck + the market gap (report §1.1–1.2)
  2. **HOW I APPROACHED IT** — the Double Diamond, stakeholder interviews with the lab
     workforce, the 19-system market survey → the innovation brief *(its own spotlight — D-09)*
  3. **THE BUILD** — the cassette→RunOne pivot, the 5 CAD parts, the OT-2 protocol
     (what it *does*: parameterised runs, air-gap dispensing, split-strip loading)
  4. **THE RUN** — the real timelapse: the OT-2 in the DTU lab with Sirio's own parts
  5. **OUTCOME + FUTURE** — 144 samples/run, ~10 min/gel, 12/12; future perspectives
- **D-08:** **Curated case study** — ~5–6 sections, scannable in ~2 min; depth present
  but selective. A recruiter gets it fast; an engineer sees the rigor.
- **D-09:** **The design-thinking process gets its OWN spotlight section** — it's the
  differentiator that signals "designer," not just "programmer" (Double Diamond,
  interviews, 19-system survey → brief).
- **D-10:** ⚠ **DROP the "~785 lines" framing everywhere.** Line count is vanity, not
  value (Sirio: *"What's the use of that?"*). Describe what the protocol **does**, never
  its length. (Supersedes the profile.md phrasings at `:102`, `:283`.)

### Motion / the "moving parts" (success criterion 4)

- **D-11:** **The real timelapse is the signature automation motion** — authentic (the
  actual OT-2 running in the DTU lab with Sirio's green 3D-printed parts). Carries
  criterion 4, paired with Phase 2's scroll-reveal vocabulary and the new GSAP
  scroll-telling.
- **D-12:** **The animated OT-2 deck is an OPTIONAL throwaway sketch, not a commitment.**
  Sirio liked the idea but is (rightly) wary of polish risk. At build time, do ONE
  throwaway `/gsd-sketch`; if it clearly lands, add it as a bonus; if not, the timelapse
  stands alone and we drop it. Upside-only, zero risk to the page. (Honest odds given to
  Sirio: motion-quality ~90%, protocol-accuracy ~80% because the data exists, first-pass
  non-gimmicky polish ~60–70% — which is exactly why it's a throwaway sketch.)
- **D-13:** **GSAP + ScrollTrigger drives the scroll-telling** — parts assembling, the
  protocol tracing itself, blueprint lines drawing on scroll. This is where "advanced"
  becomes visible wow. See Tech Direction (D-19–D-22).

### Media

- **D-14:** **Media set for the page:** `TimelapseOT2.mp4` (the star — needs a pragmatic
  <2MB re-encode + a `poster`, per D-31 of Phase 2), the **4 CAD renders** (genuine
  alpha → blueprint-plate treatment, per D-25/D-28 of Phase 2), the **Opentrons Slots
  allocation.png** (deck map; also the reference for the D-12 sketch), and the
  **UI protocol runner.png** (the parameterised-run interface — evidence of what the
  protocol does).
- **D-15:** ⚠ **Both `Opentrons Slots allocation.png` and `UI protocol runner.png` are
  genuinely opaque-white** (Phase 2 D-26; the UI shot is 371×888 portrait). They need a
  **blueprint-plate / device-frame** treatment to sit on the dark theme — not a raw drop.
- **D-16:** ⚠ **UNRESOLVED — vendor images (Phase 2 D-33).** `Opentrons-OT2.png` and
  `RunOne Gel Electrophoresis System.png` are the manufacturers' copyrighted marketing
  renders. Sirio chose **"check press-kit terms first"** → researcher task R-02. If not
  permitted for editorial use, **fall back to Sirio's own bench photo or line-art** for
  the "what I started with → what I built" contrast. A bench photo is a nice-to-gather.

### Tech direction (site-level implication — flagged to Sirio)

- **D-17:** ⚠ **"Locked to vanilla" was a mischaracterization** and is corrected. The
  actual hard constraint is **static + deployable to GitHub Pages, no server runtime**.
  Sirio wants the site to *showcase what he can build with AI*, so advanced resources are
  welcome where they earn their place.
- **D-18 [informational]:** **Chosen direction: static foundation + selective modern libraries.** Keep
  the current site and Canvas engine as-is; **no framework rewrite** (React/Next rejected —
  big cost, zero recruiter-visible payoff, fights the in-document overlay architecture).
- **D-19:** **ADD GSAP + ScrollTrigger (CDN) for the OT-2 page's scroll-telling.** GSAP +
  ScrollTrigger are free, static-safe, and the industry standard for exactly this.
- **D-20:** ⚠ **This deliberately revisits Phase 2's "no motion library" / "single rAF
  loop" notes (02-CONTEXT D-35, D-37).** Justification: the `[data-proj]` overlay
  architecture **pauses the Canvas deep-field engine while a project page is open**
  (`Sirio V4…dc.html:352` — `drawDeep`/`applyPara` are skipped when `curProj` is set), so
  GSAP can own the subpage's loop without contending with the star engine. The planner
  must reconcile this **deliberately**, not silently.
- **D-21 [informational]:** **3D (Three.js / `<model-viewer>`) = later opt-in stretch, NOT baseline.** The
  project deliberately chose Canvas-2D over WebGL for mobile lightness; a 3D showpiece is
  the overcomplication trap for a GitHub-Pages portfolio. Available if Sirio wants to go
  bigger later.
- **D-22 [informational]:** ⚠ **Project-level docs may need a light update:** `CLAUDE.md` and `PROJECT.md`
  currently frame the stack as "Canvas-2D + modern HTML/CSS/JS." The GSAP decision widens
  that to "static + selective modern libraries permitted." **Flag to Sirio; do not silently
  rewrite** the sources-of-truth.

### UI direction (design-taste-frontend applied in discussion — Sirio's explicit ask)

- **D-23 [informational]:** **The `design-taste-frontend` skill is installed and canonical for this phase.**
  Location: `.claude/skills/design-taste-frontend/SKILL.md` (source:
  https://github.com/Leonxlnx/taste-skill). Used at **build time** (plan + execute), and
  Sirio may invoke `/design-taste-frontend` himself in a sketch to steer the look.
- **D-24 [informational]:** ⚠ **Take the skill's TASTE, not its STACK.** The skill assumes
  React/Next/Tailwind/Motion/GSAP; our site is vanilla + Canvas. Its code skeletons
  (Section 5) do NOT apply. Its *design judgment* does: the design read, the dials, layout
  discipline, hero rules, "motion must be motivated," content-density and AI-tell bans.
- **D-25:** **Design Read:** *a project case-study subpage for a designer-engineer's
  portfolio, for recruiters + technical collaborators, in the site's existing dark
  "engineered bioluminescence" language — a **preserve-and-extend** of the Deep Field ×
  Hyperlight system, not a greenfield.* (Skill Section 11 redesign/preserve mode applies.)
- **D-26:** **The three dials (match the home page):** `DESIGN_VARIANCE: 8` (asymmetric
  blueprint-plate compositions), `MOTION_INTENSITY: 7` (living bg + reveals + warp +
  scroll-telling, all *motivated*), `VISUAL_DENSITY: 3–4` (curated, media-forward, airy).
- **D-27:** **Page character = "the engineering room."** The warp EARNS a shift: still dark
  and on-theme, but on arrival the **blueprint grid comes forward, the starfield recedes**,
  a more schematic/CAD character, with GSAP scroll-telling driving parts-assemble and
  protocol-trace. Dramatizes the click and suits an automation project. (NOT a seamless
  same-world continuation.)
- **D-28:** ⚠ **Override the skill's em-dash ban.** The skill bans em/en-dashes as an
  anti-AI-tell rule, but the existing site *deliberately* uses them as its terse mono-label
  voice. Preserve-mode keeps the established voice — **do NOT strip the site's punctuation**
  to satisfy the skill. Known, intentional deviation.

- **The V4 warp machinery** (`Sirio V4…dc.html:266-520`) — a complete, reduced-motion-aware
  transition system. Port it; do not reinvent. Includes the overlay canvas, the masked mid-point
  DOM swap, and the palette-aware warp tint.
- **The `[data-proj]` overlay pattern** — deep-dive pages live in the SAME document as the home,
  shown/hidden via visibility + the warp. `data-open="id"` on a card opens `[data-proj="id"]`.
- **The engine PAUSES behind an open overlay** (`:352`) — makes GSAP safe on the subpage (D-20).
- **`REDUCED`** (`star-engine.js:12`) — the single reduced-motion source of truth; GSAP must gate
  on it too (no second media-query source — 02-CONTEXT D-36).
- **`--accent` section system** (`styles.css:573-589`) — a section owns its accent by setting one
  variable; the "engineering room" may add a technical accent this way (D-27).
- **`.media-slot` + `:has()`** — striped placeholder auto-switches off when an asset is added.
- **`.nav__link` `target="_blank" rel="noopener"` pattern** — the DALSA external link must match.

### Established Patterns
- **One rAF loop on the HOME page** (`main.js:9`) — do not add a second loop to the *home*. The
  subpage overlay is the sanctioned exception (engine paused there), where GSAP owns motion.
- **Content is written from `profile.md` / the report**, never from the design's placeholder copy.
- **No layout reads in the loop/pointer handler** — keep GSAP's measurements to its own
  ScrollTrigger, not the engine's handlers.

### Integration Points
- **The OT-2 card** (`index.html:214-223`) gains `data-open="ot2"`.
- **A new `[data-proj="ot2"]` overlay** (with `[data-proj-inner]`, `[data-close]`, `data-pos`)
  is added to the document; the warp machinery + `[data-warpov]` overlay port alongside it.

  the overlay and gated on `REDUCED`.
